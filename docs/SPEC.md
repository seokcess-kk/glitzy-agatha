# Agatha (Marketing Intelligence) 기능 명세서

## 1. 프로젝트 비전

### 1.1 목적
**"어떤 마케팅이 전환으로 이어지는가?"**

다채널 광고 유입부터 전환까지 전체 퍼널을 추적하여,
마케팅 투자 대비 실제 전환 기여도를 측정하는 업종 무관 범용 마케팅 인텔리전스 대시보드.

### 1.2 핵심 가치
| 가치 | 설명 |
|------|------|
| **광고 ROI 가시화** | 어떤 광고 캠페인이 실제 전환을 만드는지 추적 |
| **리드 전환 추적** | 유입 → 전환/미전환/보류까지 결과 기반 성과 측정 |
| **채널별 비교** | Meta, Google, TikTok, Naver, Kakao, Dable 등 채널별 전환율 비교 |
| **멀티테넌트** | 여러 클라이언트를 독립적으로 관리 |
| **클라이언트 최소 부담** | 클라이언트는 전환 결과만 입력, 나머지는 에이전시/시스템이 처리 |

### 1.3 역할 분담 모델

```
┌─ 에이전시(Glitzy) 영역 ─────────────────────────────┐
│                                                       │
│  광고 집행 → 데이터 수집(자동) → 리드 분석 → 리포트  │
│  랜딩페이지 제작, 캠페인 관리, 소재 관리              │
│  순위 모니터링, 예산 관리                              │
│                                                       │
└───────────────────────────────────────────────────────┘

┌─ 클라이언트 영역 (최소한의 입력) ────────────────────┐
│                                                       │
│  리드 확인 → 결과 입력 (전환 / 미전환 / 보류)        │
│              + 전환 금액 입력                          │
│                                                       │
└───────────────────────────────────────────────────────┘

┌─ 시스템 자동 처리 ───────────────────────────────────┐
│                                                       │
│  광고 데이터 동기화, UTM 추적, 리드 유입 기록         │
│  유입 알림 발송, KPI 자동 계산, 리포트 생성           │
│                                                       │
└───────────────────────────────────────────────────────┘
```

### 1.4 대상 사용자
| 역할 | 코드 | 사용자 | 핵심 니즈 |
|------|------|--------|----------|
| **슈퍼어드민** | `superadmin` | Glitzy (마케팅 에이전시) | 전체 클라이언트 관리, 계정 생성, 성과 비교 |
| **에이전시 담당자** | `agency_staff` | 마케팅 실행사 직원 | 배정된 클라이언트 광고 운영, 성과 분석 |
| **클라이언트 관리자** | `client_admin` | 클라이언트 대표/담당자 | 자사 데이터 조회, 전환 결과 입력, 담당자 관리 |
| **클라이언트 담당자** | `client_staff` | 클라이언트 실무자 | 리드 확인, 전환 결과 입력 |

## 2. 유입 경로 정의

### 2.1 지원 채널

```
┌─────────────────────────────────────────────────────────────┐
│                    유입 경로                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  광고 클릭   │  │  전화 문의   │  │  수동 등록   │       │
│  │  (UTM 추적)  │  │  (수동 등록) │  │  (직접 입력) │       │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘       │
│         │                 │                 │                │
│         ▼                 ▼                 ▼                │
│    웹폼 제출        클라이언트 등록    클라이언트 등록         │
│                                                              │
│  ┌──────────────┐                                            │
│  │  외부 웹훅   │  클라이언트 자체 홈페이지/폼에서           │
│  │  (API 연동)  │  리드 데이터 전송                          │
│  └──────┬───────┘                                            │
│         │                                                    │
│         ▼                                                    │

### 2.2 채널별 데이터 수집

| 채널 | 수집 방법 | 추적 데이터 |
|------|----------|------------|
| **Meta Ads** | UTM + Webhook | utm_source=meta, campaign_id |
| **Google Ads** | UTM + Webhook | utm_source=google, gclid |
| **TikTok Ads** | UTM + Webhook | utm_source=tiktok |
| **Naver Ads** | UTM + Webhook | utm_source=naver |
| **Kakao Ads** | UTM + Webhook | utm_source=kakao |
| **Dable Ads** | UTM + Webhook | utm_source=dable |
| **수동 등록** | 클라이언트 직접 입력 | 유입 경로 선택 (아래 참조) |
| **외부 웹훅** | API 수신 | 클라이언트 시스템 연동 |

### 2.3 수동 등록 유입 경로

클라이언트가 리드를 수동 등록할 때 선택하는 유입 경로 (`utm_source`에 저장):

| 유입 경로 | utm_source 값 | 설명 |
|----------|--------------|------|
| 전화 문의 | `phone` | 전화로 문의가 들어온 경우 |
| 방문 문의 | `visit` | 직접 방문한 경우 |
| 소개/추천 | `referral` | 지인이나 기존 고객 소개 |
| 기타 | `other` | 직접 입력 |

### 2.4 UTM 파라미터 체계

| 파라미터 | 용도 | 예시 |
|---------|------|------|
| `utm_source` | 유입 매체 | meta, google, naver, kakao, tiktok, dable |
| `utm_medium` | 광고 유형 | cpc, display, social, organic |
| `utm_campaign` | 캠페인명 | spring_promo, summer_sale |
| `utm_content` | 광고 소재 | banner_v1, video_30s |
| `utm_term` | 검색 키워드 | (Google/Naver Ads 키워드) |

## 3. 연락처 식별 체계

### 3.1 전화번호 기반 식별

**선택 이유:**
1. 업종 무관하게 한국 시장에서 가장 보편적인 식별자
2. 유입 경로와 무관하게 수집 가능
3. 동명이인 문제 해결
4. 기존 클라이언트 시스템과 호환 용이

### 3.2 식별 흐름

```
┌─────────────────────────────────────────────────────────────┐
│ 연락처 식별 프로세스                                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  [신규 유입]                                                 │
│      │                                                       │
│      ▼                                                       │
│  전화번호 입력                                               │
│      │                                                       │
│      ▼                                                       │
│  정규화: 01012345678 → 010-1234-5678                        │
│      │                                                       │
│      ▼                                                       │
│  DB 조회: contacts.phone_number = ?                         │
│      │                                                       │
│      ├─ NOT FOUND → 신규 연락처 생성                        │
│      │              ├─ first_source 기록                    │
│      │              └─ first_campaign_id 기록               │
│      │                                                       │
│      └─ FOUND → 기존 연락처 연결                            │
│                 └─ leads에 재유입 기록                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 데이터 구조

```
contacts 테이블
├── id (PK)
├── phone_number (UNIQUE per client, 정규화됨) ← 식별자
├── name
├── first_source (최초 유입 채널)
├── first_campaign_id (최초 캠페인)
├── client_id (FK)
├── total_conversions (누적 전환 횟수)
├── total_conversion_value (누적 전환 금액)
└── created_at
```

### 3.4 예외 처리

| 상황 | 처리 방법 |
|------|----------|
| 전화번호 변경 | 관리자가 수동으로 연락처 병합 |
| 번호 없음 | 이메일 또는 임시 ID 발급 |
| 중복 문의 | 동일 번호 = 기존 연락처에 리드 추가 (재유입) |

## 4. 전환 퍼널 정의

### 4.1 심플 전환 모델

클라이언트 부담을 최소화하기 위해 **리드 상태 = 퍼널 단계**인 심플 모델을 채택한다.

```
┌─────────────────────────────────────────────────────────────┐
│ 전환 퍼널                                                    │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                                            │
│  │  신규(New)  │  광고 클릭 / 전화 / 웹훅 / 수동 등록       │
│  │             │  → leads 테이블 자동 생성                   │
│  └──────┬──────┘                                            │
│         │                                                    │
│         ▼                                                    │
│  ┌──────────────────┐                                       │
│  │ 진행중             │  클라이언트가 리드 확인 후            │
│  │ (In Progress)     │  연락 시도 / 상담 진행 중             │
│  └──────┬───────────┘                                       │
│         │                                                    │
│         ├──────────────┬──────────────┐                     │
│         ▼              ▼              ▼                     │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐              │
│  │   전환    │  │  미전환   │  │   보류    │              │
│  │(Converted)│  │  (Lost)   │  │  (Hold)   │              │
│  │+ 전환금액 │  │ + 사유    │  │→ 추후재확인│              │
│  └───────────┘  └───────────┘  └───────────┘              │
│                                                              │
│  ※ 무효(Invalid): 스팸/중복/잘못된 번호 — 에이전시가 처리  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**퍼널 시각화 (대시보드):**
```
New(유입) → In Progress(진행) → Converted(전환)
  150건         85건              45건
          (57%)           (53%)
```

### 4.2 리드 상태 정의

| 상태 | 코드 | 설명 | 입력 주체 |
|------|------|------|-----------|
| 신규 | `new` | 유입 직후, 아직 확인 안 됨 | 시스템 자동 |
| 진행중 | `in_progress` | 연락 시도 / 상담 진행 중 | 클라이언트 |
| 전환 | `converted` | 구매/계약/등록 완료 + 전환 금액 | 클라이언트 |
| 보류 | `hold` | 아직 결정 안 됨 (추후 재확인) | 클라이언트 |
| 미전환 | `lost` | 거절/이탈 + 사유 | 클라이언트 |
| 무효 | `invalid` | 잘못된 연락처, 스팸, 중복 | 에이전시 |

**클라이언트의 일반적인 흐름:**
1. 신규 리드 알림 수신
2. "진행중" 처리 (내가 확인했고 연락 중이다)
3. 결과 입력: "전환" (+ 금액) 또는 "미전환" (+ 사유) 또는 "보류"

### 4.3 미전환 사유 (선택 입력)

| 사유 | 코드 | 설명 |
|------|------|------|
| 연락 안 됨 | `no_response` | 전화 부재, 무응답 |
| 관심 없음 | `not_interested` | 고객이 관심 없다고 응답 |
| 가격 문제 | `price_issue` | 가격이 맞지 않음 |
| 경쟁사 선택 | `chose_competitor` | 다른 업체 선택 |
| 시기 아님 | `bad_timing` | 지금은 아니지만 나중에 |
| 기타 | `other` | 직접 입력 |

### 4.4 전환 입력 항목

| 항목 | 필수 | 설명 |
|------|------|------|
| 전환 상태 | 필수 | in_progress / converted / lost / hold |
| 전환 금액 | 전환 시 필수 | 실제 계약/결제 금액 |
| 메모 | 선택 | 자유 텍스트 |
| 미전환 사유 | 미전환 시 선택 | 위 사유 목록에서 선택 |

## 5. KPI 계산 명세

### 5.1 핵심 지표

**광고 효율 지표:**

| 지표 | 계산식 | 단위 | 용도 |
|------|--------|------|------|
| **CPL** | 광고비 ÷ 리드수 | 원 | 리드 획득 비용 |
| **ROAS** | (전환 금액 합계 ÷ 광고비) × 100 | % | 광고 투자 수익률 |
| **CAC** | 총광고비 ÷ 전환 고객수 | 원 | 고객 획득 비용 |
| **CVR** | 전환수 ÷ 리드수 × 100 | % | 리드 전환율 |
| **CPC** | 광고비 ÷ 클릭수 | 원 | 클릭당 비용 |
| **CTR** | 클릭수 ÷ 노출수 × 100 | % | 클릭률 |

**전환 분석 지표:**

| 지표 | 계산식 | 단위 | 용도 |
|------|--------|------|------|
| **거부율** | 미전환 수 ÷ 전체 유효 리드 × 100 | % | 리드 품질/상품 경쟁력 평가 |
| **보류율** | 보류 수 ÷ 전체 유효 리드 × 100 | % | 잠재 전환 규모 파악 |
| **노쇼율** | 연락 안 됨(no_response) 수 ÷ 전체 유효 리드 × 100 | % | 접촉 실패율 |
| **무효율** | 무효 리드 수 ÷ 전체 리드 × 100 | % | 리드 품질 (스팸/중복 비율) |
| **유효 리드율** | 유효 리드 ÷ 전체 리드 × 100 | % | 유효 DB 비율 |
| **보류→전환율** | 보류 후 전환 수 ÷ 보류 수 × 100 | % | 재접촉 효과 측정 |

### 5.2 집계 기준

모든 KPI는 다음 기준으로 집계 가능하며, **각 기준별로 전환율, 거부율, 보류율이 함께 표시**된다:

| 기준 | 설명 | 예시 |
|------|------|------|
| **전체** | 전체 데이터 집계 | 이번 달 전체 ROAS |
| **채널별** | utm_source 기준 | Meta ROAS vs Google ROAS |
| **캠페인별** | utm_campaign 기준 | spring_promo CPL |
| **소재별** | utm_content 기준 | banner_v1 vs video_30s CTR |
| **랜딩페이지별** | landing_page_id 기준 | LP-A 전환율 vs LP-B 전환율 |
| **기간별** | 일/주/월/분기 | 주간 리드 추이 |
| **클라이언트별** | client_id 기준 | A사 vs B사 성과 |

### 5.3 기준별 전환율 테이블 예시

```
┌──────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ 기준     │ 리드   │ 전환   │ 전환율 │ 거부율 │ 보류율 │ 노쇼율 │
├──────────┼────────┼────────┼────────┼────────┼────────┼────────┤
│ Meta     │  80건  │  25건  │ 31.3%  │ 25.0%  │ 12.5%  │ 18.8%  │
│ Google   │  40건  │  12건  │ 30.0%  │ 30.0%  │ 10.0%  │ 15.0%  │
│ Naver    │  20건  │   5건  │ 25.0%  │ 35.0%  │ 15.0%  │ 20.0%  │
│ LP-메인  │  90건  │  28건  │ 31.1%  │ 27.8%  │ 11.1%  │ 17.8%  │
│ LP-이벤트│  50건  │  14건  │ 28.0%  │ 30.0%  │ 14.0%  │ 18.0%  │
└──────────┴────────┴────────┴────────┴────────┴────────┴────────┘
```

### 5.4 예산 관련 지표

| 지표 | 계산식 | 용도 |
|------|--------|------|
| **예산 소진율** | 현재 지출 ÷ 월 예산 × 100 | 예산 관리 |
| **일 평균 소진** | 월 누적 지출 ÷ 경과 일수 | 소진 속도 추정 |
| **예상 월말 지출** | 일 평균 소진 × 해당 월 일수 | 예산 초과 예측 |

## 6. 기능별 상세 스펙

### 6.1 광고 성과 분석

**목적**: 다채널 광고의 성과를 통합 분석

**주요 화면 구성:**
- **플랫폼 요약**: 채널별 지출/노출/클릭/리드/전환 한눈에 비교
- **캠페인 분석**: 캠페인별 상세 성과 (CPL, ROAS, CVR)
- **소재 성과**: 광고 소재별 클릭률, 전환율 비교
- **효율 추이**: 일별/주별 CPL, ROAS 트렌드 차트
- **요일별 분석**: 요일별 광고 성과 패턴
- **랜딩페이지 분석**: 랜딩페이지별 전환 성과 비교

**데이터 소스**: 광고 플랫폼 API 자동 동기화 (일 1회, 매일 06:00 KST) + 수동 동기화 버튼 (즉시 실행)

### 6.2 캠페인 관리

**목적**: 클라이언트별 캠페인 메타데이터 관리

**기능:**
- 캠페인 목록 조회 (플랫폼별 필터)

### 6.3 광고 소재 관리

**목적**: 광고 크리에이티브 자산 관리 및 성과 분석

**기능:**

- 소재별 성과 분석 (노출, 클릭, CTR, 전환, ROAS)
- 소재 간 A/B 비교

### 6.4 랜딩페이지 빌더

**목적**: 리드 수집용 랜딩페이지 제작 및 관리

**기능:**

### 6.6 고객관리

**목적**: 유입된 리드의 전환 결과 추적 및 고객 DB 관리

하나의 페이지에서 **리드 탭**과 **고객DB 탭**으로 구분한다.

| 구분 | 리드 탭 | 고객DB 탭 |
|------|---------|-----------|
| **단위** | 유입 건 (1명이 3번 문의 = 3건) | 사람 (1명 = 1건) |
| **주 용도** | 전환 결과 입력, 일상 운영 | 고객 이력 조회, 재유입 분석 |
| **핵심 정보** | 유입 채널, UTM, 상태, 전환 금액 | 총 문의 수, 총 전환 금액, 최초 채널 |
| **사용 빈도** | 매일 (실무) | 가끔 (분석) |

**리드 탭 기능:**
- 리드 목록 (필터: 상태, 채널, 기간, 캠페인, 랜딩페이지)
- 리드 상세 (유입 정보, UTM, 연락처, 전환 이력)
- 전환 결과 입력 (진행중/전환/미전환/보류 + 금액 + 사유)
- 리드 수동 등록 (전화 문의 등 — 클라이언트가 직접 입력)
- 리드 내보내기 (CSV)
- 리드 중복 감지 (전화번호 기반)

**고객DB 탭 기능:**
- 고객(연락처) 목록 (이름, 전화번호, 최초 유입 채널, 총 전환 금액)
- 고객 상세 (유입 이력, 전환 이력 타임라인)
- 신규 vs 재유입 구분
- 고객별 리드 목록 연결

**클라이언트 화면:**
```
┌─────────────────────────────────────────────────────────────┐
│ 고객관리                           [이번 달 ▼] [+ 수동 등록]│
├─────────────────────────────────────────────────────────────┤
│  [리드]  [고객DB]                                            │
│                                                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │ 전체    │  │  신규   │  │  진행중  │  │  전환   │       │
│  │  150건  │  │   32건  │  │   48건  │  │   45건  │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘       │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ 이름     │ 연락처       │ 유입채널 │ 상태    │ 결과  │   │
│  ├──────────┼──────────────┼──────────┼─────────┼───────┤   │
│  │ 홍길동   │ 010-1234-**** │ Meta     │ 신규   │ [입력]│   │
│  │ 김철수   │ 010-5678-**** │ Google   │ 전환   │ 50만  │   │
│  │ 이영희   │ 010-9012-**** │ Naver    │ 진행중 │ [입력]│   │
│  └──────────┴──────────────┴──────────┴─────────┴───────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 6.7 유입 알림

**목적**: 클라이언트별 광고 예산 설정 및 소진 추적

**기능:**
- 월별 광고 예산 설정 (클라이언트별)
- **예산 수정 이력 관리** — 중간에 캠페인/상품 추가 시 당월 예산 증액 가능
  - 변경 시 사유 메모 기록

  - 월별 예산 추이 분석
- 예산 소진율 실시간 추적
- 예산 초과 임박 시 알림 (80%, 100% 도달)

**예산 수정 예시:**
```

**커스터마이징 (superadmin 설정, 클라이언트별):**

- 활성화/비활성화 토글
- Cron이 매일 체크 → 해당일에 발송 예정인 리포트만 발송

### 6.12 계정 관리 (초대 기반 회원가입)

**목적**: 클라이언트 온보딩 간소화 및 대규모 계정 관리

**두 가지 계정 생성 방식 병행:**

| 방식 | 용도 | 생성 주체 |
|------|------|-----------|
| 관리자 직접 생성 | 초기 세팅, 긴급 계정 | superadmin |
| 초대 링크 발급 | 클라이언트 온보딩 기본 방식 | superadmin, client_admin |

**초대 흐름:**
```
1. 에이전시가 초대 생성
   → 클라이언트, 역할, 만료일 지정
   → 초대 링크(토큰) 생성

2. 초대 링크 전달
   → 카카오톡/SMS/이메일로 발송

3. 클라이언트가 링크로 접속
   → 회원가입 폼 (이름, 휴대폰 번호, 비밀번호)
   → 휴대폰 번호 = 로그인 아이디 + 알림 수신 번호
   → 번호 유효성 검증 (010 시작, 11자리)
   → 클라이언트 + 역할 자동 매핑

4. 가입 완료
   → 즉시 로그인 가능
```

**초대 권한:**
- superadmin → agency_staff 초대
- superadmin → client_admin 초대
- client_admin → client_staff 초대 (자기 클라이언트 내에서)

**초대 관리:**
- 초대 목록 조회 (대기중/완료/만료)
- 초대 취소/재발송
- 초대 만료: 기본 7일, 설정 가능

### 6.13 데모 모드

**목적**: 신규 클라이언트 영업 시 샘플 데이터로 대시보드 시연

**기능:**
- 데모 전용 계정 (`demo_viewer` 역할)
- 미리 생성된 샘플 데이터 (리드, 광고 성과, 전환 등)
- 읽기 전용 (데이터 수정 불가)
- 제한된 페이지 접근 (핵심 기능만)

### 7.2 핵심 테이블 스키마

#### clients (클라이언트)
```sql
CREATE TABLE clients (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  slug VARCHAR(50) UNIQUE,
  monthly_budget DECIMAL(12,2),            -- 월 광고 예산
  is_active BOOLEAN DEFAULT TRUE,
  notify_phones TEXT[],                    -- 알림 수신 전화번호 목록
  created_at TIMESTAMP DEFAULT NOW()
);
```

#### contacts (연락처)
```sql
CREATE TABLE contacts (
  id SERIAL PRIMARY KEY,
  client_id INTEGER REFERENCES clients(id),
  phone_number VARCHAR(20) NOT NULL,       -- 정규화된 전화번호 (식별자)
  name VARCHAR(100),
  first_source VARCHAR(50),                -- 최초 유입 채널
  first_campaign_id VARCHAR(100),          -- 최초 캠페인
  total_conversions INTEGER DEFAULT 0,     -- 누적 전환 횟수
  total_conversion_value DECIMAL(12,2) DEFAULT 0, -- 누적 전환 금액
  created_at TIMESTAMP DEFAULT NOW(),

  UNIQUE(client_id, phone_number)          -- 클라이언트별 고유
);
```

#### leads (리드)
```sql
CREATE TABLE leads (
  id SERIAL PRIMARY KEY,
  client_id INTEGER REFERENCES clients(id),
  contact_id INTEGER REFERENCES contacts(id),

  -- UTM 파라미터
  utm_source VARCHAR(50),
  utm_medium VARCHAR(50),
  utm_campaign VARCHAR(100),
  utm_content VARCHAR(100),
  utm_term VARCHAR(100),

  -- 유입 정보
  landing_page_id INTEGER REFERENCES landing_pages(id),
  inflow_url TEXT,

  -- 전환 추적
  status VARCHAR(20) DEFAULT 'new',        -- new/in_progress/converted/lost/hold/invalid
  lost_reason VARCHAR(30),                 -- 미전환 사유
  conversion_value DECIMAL(12,2),          -- 전환 금액
  conversion_memo TEXT,                    -- 메모

  -- 타임스탬프
  status_changed_at TIMESTAMP,             -- 상태 변경 시각
  created_at TIMESTAMP DEFAULT NOW(),
  updated_by INTEGER REFERENCES users(id)
);
```

#### ad_campaign_stats (광고 캠페인 통계)
```sql
CREATE TABLE ad_campaign_stats (
  id SERIAL PRIMARY KEY,
  client_id INTEGER REFERENCES clients(id),
  platform VARCHAR(20),                    -- meta/google/tiktok/naver/kakao/dable
  campaign_id VARCHAR(100),
  campaign_name VARCHAR(200),
  campaign_type VARCHAR(50),               -- 캠페인 유형 (feed, reels, search 등)
  stat_date DATE,
  impressions INTEGER DEFAULT 0,
  clicks INTEGER DEFAULT 0,
  spend_amount DECIMAL(12,2) DEFAULT 0,
  conversions INTEGER DEFAULT 0,
  created_at TIMESTAMP DEFAULT NOW(),

  UNIQUE(client_id, platform, campaign_id, stat_date)
);
```

#### landing_pages (랜딩페이지)
```sql
CREATE TABLE landing_pages (
  id SERIAL PRIMARY KEY,
  client_id INTEGER REFERENCES clients(id),
  random_id VARCHAR(8) UNIQUE,             -- 8자리 랜덤 ID
  title VARCHAR(200),
  gtm_id VARCHAR(50),                      -- GTM 컨테이너 ID
  redirect_url TEXT,                       -- 전환 후 리다이렉트
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT NOW()
);
```

#### users (사용자)
```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  phone_number VARCHAR(20) UNIQUE NOT NULL, -- 로그인 아이디 (휴대폰 번호)
  password_hash VARCHAR(200) NOT NULL,
  name VARCHAR(100) NOT NULL,
  role VARCHAR(20) NOT NULL,                -- superadmin/agency_staff/client_admin/client_staff/demo_viewer
  client_id INTEGER REFERENCES clients(id), -- client_admin/client_staff는 필수
  is_active BOOLEAN DEFAULT TRUE,
  password_version INTEGER DEFAULT 1,
  created_at TIMESTAMP DEFAULT NOW()
);
```

#### invitations (초대)
```sql
CREATE TABLE invitations (
  id SERIAL PRIMARY KEY,
  token VARCHAR(100) UNIQUE NOT NULL,       -- 초대 링크 토큰
  client_id INTEGER REFERENCES clients(id), -- 초대할 클라이언트
  role VARCHAR(20) NOT NULL,                -- 부여할 역할
  invited_by INTEGER REFERENCES users(id),  -- 초대한 사람
  expires_at TIMESTAMP NOT NULL,            -- 만료일 (기본 7일)
  status VARCHAR(20) DEFAULT 'pending',     -- pending/completed/expired/cancelled
  completed_by INTEGER REFERENCES users(id),-- 가입 완료한 사용자
  completed_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);
```

## 11. 검증 기준

SPEC 구현 완료 시 체크리스트:

- [ ] Meta 광고 캠페인의 리드 유입을 UTM으로 추적 가능
- [ ] 클라이언트가 고객관리 > 리드 탭에서 진행중/전환/미전환/보류 결과 입력 가능
- [ ] 클라이언트가 리드를 수동 등록(전화 문의 등) 가능
- [ ] 전환 금액 입력 후 ROAS가 자동 계산됨
- [ ] 채널별(Meta/Google/Naver 등) CPL, ROAS, 전환율, 거부율, 보류율 비교 가능
- [ ] 랜딩페이지별 전환율, 거부율 비교 가능
- [ ] 캠페인별 성과 분석 가능
- [ ] 소재별 CTR, 전환율 비교 가능
- [ ] 퍼널 시각화: New → In Progress → Converted + 보류/미전환 비율 표시
- [ ] 신규 리드 유입 시 담당자에게 카카오톡/SMS 알림 발송됨
- [ ] 월 예산 설정 및 중간 수정(증액) 가능, 변경 이력 조회 가능
- [ ] 예산 소진율 추적 및 초과 알림 동작
- [ ] 광고 데이터 자동 동기화(06:00 KST) + 수동 동기화 버튼 동작
- [ ] 랜딩페이지로 유입된 리드가 자동 생성됨
- [ ] UTM 링크 + QR코드 생성 가능
- [ ] 순위 모니터링 키워드의 일별 순위 추적 가능
- [ ] 멀티테넌트 데이터 격리가 정상 동작 (클라이언트 간 데이터 누출 없음)
- [ ] 4개 역할의 접근 제어가 정상 동작
- [ ] CSV로 리드 데이터 내보내기 가능
- [ ] 고객DB 탭에서 고객별 전환 이력 타임라인 조회 가능
- [ ] 초대 링크로 회원가입 시 클라이언트 + 역할 자동 매핑 가능
- [ ] client_admin이 자기 클라이언트 내 client_staff 초대 가능
- [ ] 데모 모드에서 샘플 대시보드 확인 가능
